WF4 Publisher — 3 Approaches

How should the publishing workflow be structured in n8n?

A

Single workflow, all 6 platforms ⭐ Recommended

One WF4 workflow handles everything: receives the approval webhook, then posts to all 6 platforms in sequence. Each platform node has "Continue on Fail" so if Instagram fails, it still posts to X, LinkedIn, etc.

Approve → WF4 webhook → Get Draft → Post IG → Post FB → Post WA → Post X → Post LI → Post PIN → Update DB ✓

Pros

  • Simplest to build
  • Easy to debug
  • Matches WF1–3 style

Cons

  • Platforms run one at a time
B

Orchestrator + 6 sub-workflows

WF4 is a "router" that fires 6 child webhooks in parallel — one per platform. Each platform lives in its own mini-workflow you can turn on/off independently.

Approve → WF4 → WF4a(IG) + WF4b(FB) + WF4c(X) … (parallel)

Pros

  • Platforms run in parallel
  • Toggle platforms on/off

Cons

  • 7 workflows to maintain
  • Complex for a first setup
C

Phased — Meta first, then expand

Build WF4 with only Meta platforms first (IG + FB + WhatsApp — one set of API credentials). Then add X, LinkedIn, Pinterest as you get their credentials set up.

Phase 1: Approve → WF4 → Post IG + FB + WA
Phase 2: + Post X + LI + PIN (add nodes later)

Pros

  • Get something working fast
  • Less credential setup upfront

Cons

  • Still rewrite WF4 later
  • Slower to full coverage
